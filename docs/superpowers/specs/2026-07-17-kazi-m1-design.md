M1 delivers local routing: every HTTP service reachable at `https://<stack>.localhost` with a green lock, stable opt-in host ports for raw TCP services, and `kazi urls` as the single answer to "where is everything?"

## Decisions locked during design

| Decision | Choice |
|---|---|
| Proxy | **Caddy** (`caddy:2-alpine`). kamal-proxy evaluated and rejected for M1: no internal CA (kazi would have to become its own mkcert for `*.localhost` certs) and its health-gated deploy model fits production, not half-up dev stacks. Measured on-machine: caddy 84.6MB image / ~11MiB idle RSS vs kamal-proxy 171MB / ~3MiB ā€” weight is a non-factor. kamal-proxy noted as a candidate M4 backend. |
| Proxy connectivity | **ONCE-style shared network**: one external network `kazi`; the injected override attaches routed services to it. No socket-watching (kazi renders config itself), no per-up `network connect` choreography. |
| Hostnames | Primary + service subdomains: `<stack>.localhost` ā†’ primary HTTP service; every other HTTP service gets `<service>.<stack>.localhost`. |
| TCP host ports | **Opt-in only.** kazi never publishes a port the compose file doesn't. `kazi expose` is the one-command opt-in; discovery nudges but never acts. |
| Audience | TUI (M5) will be the primary human interface. The CLI is for occasional debugging and for agents: stable `--json` schemas and scriptability outrank human ergonomics, and no capability may exist only in a skin ā€” everything goes through the engine. |

## The `kazi-proxy` system stack

kazi-proxy is itself a compose stack ā€” generated by kazi into `~/.config/kazi/proxy/`, registered with `system: true` (visible in `status`, protected from `kazi rm`):

- `compose.yml`: `caddy:2-alpine`, publishes host `80`/`443`, joins the external `kazi` network, mounts the generated `Caddyfile` read-only and a named volume at `/data`. The volume holds Caddy's internal CA, so certificates ā€” and the trust installed by `kazi trust` ā€” survive proxy recreation.
- `Caddyfile`: fully generated, never hand-edited (header comment states this). One site block per hostname; `tls internal` (Caddy's local CA handles `*.localhost` automatically).

The `kazi` network and the proxy are created lazily on the first `up` of a stack with a routable service. No daemon at rest.

## Connectivity

The M0 override file (which already injects `kazi.*` labels) grows one concern: services classified as routable are attached to the external `kazi` network alongside their stack-private networks (compose merges service networks by union; stack-internal DNS is untouched).

**Alias collision rule:** two stacks may both have a service named `web`, and plain service aliases would collide on the shared network. The override therefore sets a *network-scoped* alias `<service>.<stack>` on the `kazi` network, and the Caddyfile dials that: `reverse_proxy web.blog:8080`.

Consequence: **only stacks kazi has up'ed get routes.** A discovered stack appears in `ls` immediately but joins the proxy after its first `kazi up` (the same moment it gets labels).

## Generate-and-reload loop

On every state-changing verb (`up`, `down`, `restart`, `rm`):

1. Engine recomputes desired routes from its typed stack view (the M0 seam).
2. Renders the Caddyfile. Unchanged content ā†’ stop.
3. Validates inside the container (`caddy validate`); a render bug leaves the old config serving.
4. Writes and runs `<runtime> exec kazi-proxy caddy reload` ā€” graceful, zero-downtime. Starts the proxy first if needed.

A proxy failure **never blocks the stack operation**: `kazi up blog` succeeds even if reload fails; the error surfaces as a warning and in `kazi status`.

## Service classification & routing rules

Optional declaration in the Stack manifest:

```yaml
spec:
  proxy:
    service: web        # primary HTTP service (optional)
    http_port: 8080     # its container port (optional if the service exposes exactly one)
    enabled: true       # false opts the whole stack out of routing
```

Auto-detection when undeclared, driven by lists in `config.yaml` (`spec.proxy.tcpPorts` seeded with 5432, 6379, 3306, 27017, 5672, 9092, ā€¦; `spec.proxy.httpPorts` seeded with 80, 8080, 3000, 8000, 5173, ā€¦):

- **HTTP service:** exposes a known HTTP port, or exactly one port not on the TCP list.
- **TCP service:** exposed ports are on the TCP list.
- **Primary:** the declared service, else the stack's *single* HTTP service. Multiple HTTP services with no declaration ā‡’ no bare `<stack>.localhost`; each service still gets `<service>.<stack>.localhost` and `kazi urls` prints the exact `spec.proxy` declaration to add. Deterministic ā€” no name-guessing.

## Stable TCP ports (opt-in)

**Boundary: the compose file is the user's declared surface.** Ports already published there are respected and recorded for `kazi urls`; kazi adds nothing on its own.

Opt-in is one command, and it is declarative ā€” it writes user-owned intent into the manifest:

```
kazi expose blog postgres              # stable port from the range
kazi expose blog postgres --port 5433  # pinned
kazi expose --remove blog postgres
```

```yaml
spec:
  expose:
    - service: postgres   # container port auto-detected if the service exposes exactly one
      port: auto          # or a fixed number
```

- Allocation from `config.yaml` `spec.ports.range` (default `42000-42999`); bookkeeping in engine-owned `~/.config/kazi/state/ports.yaml` ā€” not the manifest (spec is user-owned, status is never persisted, allocations are engine bookkeeping).
- Stable per `(stack, service, container-port)`; survives `down`; freed on `kazi rm` or `expose --remove`.
- Injected via the same override file (`ports: "42017:5432"`); if the stack is running, only the affected service is restarted.
- Discovery nudges, never acts: a known-TCP service without a host binding is shown by `urls`/`status` as unreachable, with the exact `kazi expose` command to change that.

## `kazi urls` and `kazi trust`

- **`kazi urls [stack]`** ā€” every endpoint, both kinds, `--json` included: `https://blog.localhost ā†’ web:8080`, `localhost:42017 ā†’ blog/postgres:5432`, plus unreachable-endpoint nudges.
- **`kazi trust [--uninstall]`** ā€” extracts the root CA from the proxy volume (`<runtime> exec kazi-proxy cat /data/caddy/pki/authorities/local/root.crt`) and installs it into the system store. macOS first-class: `security add-trusted-cert` into the System keychain (one sudo prompt). Linux best-effort: `update-ca-certificates`. Run once per machine ā€” the CA persists in the volume. Documented caveat: Firefox trusts its own store (`security.enterprise_roots.enabled`).

## Schema & surface deltas

- **Stack manifest:** optional `spec.proxy` and `spec.expose` (above). Additive to `v1alpha1`; M0 manifests remain valid.
- **Config:** `spec.proxy.tcpPorts`, `spec.proxy.httpPorts`, `spec.ports.range`.
- **New engine-owned paths:** `~/.config/kazi/proxy/`, `~/.config/kazi/state/ports.yaml`.
- **New commands:** `urls`, `trust`, `expose`. **Changed:** `up/down/restart/rm` end with renderā†’diffā†’validateā†’reload; `status` includes the proxy system stack.
- **M0 amendment:** stack names constrained to DNS-label form (`[a-z0-9-]`, max 63 chars) at `add` time, since names become hostnames. Fold into the M0 implementation.

## Error handling

- Proxy/reload failures: warnings, never block stack verbs; visible in `status`.
- Render safety: `caddy validate` before reload; invalid render keeps the previous config and logs the diff.
- Ports: pinned port already bound ā†’ hard error naming the conflict; range exhausted ā†’ error suggesting a wider `spec.ports.range`.
- Trust: sudo declined ā†’ non-fatal; HTTPS serves with an untrusted-cert browser warning until rerun.
- Resolution caveat: `*.localhost` works in browsers per RFC 6761; some CLI resolvers don't ā€” `urls` output includes `curl --resolve` hints. A `doctor` check lands in M3.

## Testing

- **Unit (fake runtime, golden files):** Caddyfile renderer against stack-view fixtures (including the same-service-name collision case and multi-HTTP subdomain case); port allocator (stability, pinning, conflict, exhaustion); HTTP/TCP classification table tests; override-file generation (labels + network alias + port bindings together).
- **Integration (real Docker, `integration` tag):** two stacks with identically-named `web` services both routable; `expose` round-trip including `--remove` and survival across `down`/`up`; proxy-killed-during-`up` warning path. `trust` is manual (sudo).

## Acceptance criteria

1. Fresh path: first `up` auto-creates the `kazi` network and proxy; one `kazi trust`; browser shows a green lock on `https://<stack>.localhost`.
2. Ten stacks up ā†’ every HTTP service reachable by name over HTTPS, zero memorized ports.
3. Two stacks with a same-named service route to the correct containers.
4. A multi-HTTP stack follows the primary/subdomain rules, including the no-primary hint.
5. Nothing is host-published without opt-in; `kazi expose blog postgres` yields a stable port surviving `down`/`up`, removed by `--remove`.
6. Killing the proxy container doesn't break `kazi up`; the failure appears as a warning and in `status`.
